    What‚Äôs happening:

    • Bitnami‚Äôs nginx chart pulls its container image (bitnami/nginx) from Docker Hub.

    • You got 429: toomanyrequests = rate limit exceeded.

    This is the cleanest fix.

    docker login

    (or if only kubectl/helm is available):

    kubectl create secret docker-registry dockerhub-secret \

  --docker-password=<your-password> \
  --docker-email=<your-email> \
  -n cd-tool-apd

    Then patch your ServiceAccount or Helm values to use this secret for pulling images.

    Option 2: Use Bitnami‚Äôs own container registry (quay.io or ghcr.io)¶

    Nice, you shared the chart üôå ‚Äî now it‚Äôs crystal clear why the Deployment comes out as webpage-server-02:

    In deployment.yaml, the name is set like this:

    metadata:
  name: {{ .Values.name }}

    ‚Ä¶and in values.yaml, you‚Äôve got:

    name: webpage-server-02

    So Helm is simply substituting the value, which explains the mismatch.

    üîß How to tie it to the release name¶

    Option 1 ‚Äî simplest (exam friendly) Change the deployment.yaml line to:

    metadata:
  name: {{ .Release.Name }}

    Now, when you run:

    helm upgrade --install webpage-server-01 /root/new-version

    Your Deployment will be named webpage-server-01.

    Option 2 ‚Äî values-driven (best practice) Instead of hardcoding name in values.yaml, use Helm‚Äôs built-in fullnameOverride.

    1. In values.yaml, add:
    fullnameOverride: ""
    1. In deployment.yaml, replace:
    name: {{ .Values.name }}

    with:

    name: {{ include "new-version.fullname" . }}

    ‚Ä¶and in _helpers.tpl (if your chart has it), fullname is usually defined like this:

    {{- define "new-version.fullname" -}}
{{- if .Values.fullnameOverride }}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" }}
{{- else }}
{{- .Release.Name }}
{{- end }}
{{- end }}

    This way:

    • By default, the Deployment will follow the release name (webpage-server-01).
    • If you want to override, you just set fullnameOverride in values.yaml.

    Exactly üíØ

    If you switch to:

    metadata:
  name: {{ .Release.Name }}

    üëâ then you don‚Äôt need to add anything in values.yaml. Release.Name is a built-in Helm object ‚Äî it‚Äôs always available, no matter what‚Äôs in values.yaml.

    So:

    • {{ .Values.* }} ‚Üí pulls from your values.yaml (or from --set, -f).
    • {{ .Release.Name }} ‚Üí comes directly from Helm itself (whatever name you used in helm install <release-name> ...).

    That‚Äôs why it‚Äôs often the easiest fix: you just tie the Deployment (and Service, ConfigMap, etc.) names to the release name, and you don‚Äôt have to maintain a name: field in values.yaml at all.

    üëâ Helm also ships with helpers like fullnameOverride and nameOverride (defined in _helpers.tpl when you create a chart with helm create). Those are not built-in, they‚Äôre just conventions that the starter chart scaffolds for you.
