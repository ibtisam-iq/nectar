    üîÅ Understanding Rewrite Function in Ingress Controllers (NGINX vs Traefik)¶

    üß© Overview¶

    The rewrite function in Ingress controllers controls how the request path is forwarded to the backend service.
    It doesn‚Äôt change how Kubernetes routes traffic ‚Äî it changes what path the backend application receives.

    If you hit /something, rewrite decides whether the backend receives /something or /.

    ‚öôÔ∏è 1Ô∏è‚É£ NGINX Ingress Controller¶

    üß† Default behavior¶

    NGINX forwards the full path exactly as the client sends it.

    If your Ingress rule looks like this: 

    rules:
- host: example.com
  http:
    paths:
    - path: /app
      pathType: Prefix
      backend:
        service:
          name: webapp
          port:
            number: 80

    and you access:

    curl -H "Host: example.com" http://<IP>/app

    then NGINX forwards the request to backend as:

    /app

    If the backend doesn‚Äôt have /app defined, it returns 404.

    üí° Adding Rewrite Target¶

    To fix that, you add:

    nginx.ingress.kubernetes.io/rewrite-target: /

    Now NGINX rewrites every incoming /app to / before sending it to the backend.

    ‚úÖ Backend receives / ‚úÖ Serves index.html successfully

    In short:

    In NGINX ingress, rewrite-target fixes the request path so the backend understands it.

    ‚öôÔ∏è 2Ô∏è‚É£ Traefik Ingress Controller¶

    üß† Default behavior¶

    Traefik always passes the path exactly as matched ‚Äî it does not rewrite or strip anything by default.

    If you use the same rule:

    rules:
- host: example.com
  http:
    paths:
    - path: /app
      pathType: Prefix
      backend:
        service:
          name: webapp
          port:
            number: 80

    and you hit:

    curl -H "Host: example.com" http://<IP>/app

    then backend receives /app.

    If backend doesn‚Äôt serve /app, you‚Äôll see a 404 from the backend, not from Traefik.

    üí° Rewriting in Traefik (Middleware)¶

    Traefik doesn‚Äôt support the rewrite-target annotation. Instead, it uses a Middleware resource for rewriting or stripping prefixes.

    Example:

    apiVersion: traefik.io/v1alpha1
kind: Middleware
metadata:
  name: strip-app-prefix
spec:
  stripPrefix:
    prefixes:
      - /app
---
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: webapp-ingress
  annotations:
    traefik.ingress.kubernetes.io/router.middlewares: default-strip-app-prefix@kubernetescrd
spec:
  ingressClassName: traefik
  rules:
  - host: example.com
    http:
      paths:
      - path: /app
        pathType: Prefix
        backend:
          service:
            name: webapp
            port:
              number: 80

    ‚úÖ Traefik strips /app ‚úÖ Backend receives / ‚úÖ Application loads perfectly

    üß© 3Ô∏è‚É£ Side-by-Side Summary¶

    Feature NGINX Ingress Traefik Ingress
    Rewrite mechanism Annotation ‚Üí nginx.ingress.kubernetes.io/rewrite-target Middleware ‚Üí stripPrefix
    Behavior without rewrite Passes full path Passes full path
    Default rewrite needed? Yes, for subpaths No, optional
    Backend sees /app unless rewritten /app unless middleware used
    Common issue 404 if backend has only / 404 if backend has only /
    Fix Add rewrite-target Add stripPrefix middleware

    üß† 4Ô∏è‚É£ Key Takeaways¶

    1. Rewrite doesn‚Äôt affect routing ‚Äî it affects what path the backend sees.
    2. NGINX requires a rewrite annotation when path ‚â† /.
    3. Traefik ignores rewrite-target; use a Middleware instead.
    4. If you get a 404 with HTML, it‚Äôs from backend (rewrite issue).
    5. If you get a plain 404 text, it‚Äôs from ingress (rule mismatch).

    ‚úÖ TL;DR¶

    Ingress Class Rewrite Config When Needed
    nginx nginx.ingress.kubernetes.io/rewrite-target: / For sub-paths
    traefik Middleware with stripPrefix Optional, for sub-paths

    ‚ÄúIn NGINX, rewrites live in annotations. In Traefik, rewrites live in middlewares.‚Äù
