    Create User

    üö® Error You Got:¶

    Error from server (BadRequest): CertificateSigningRequest in version "v1" cannot be handled as a CertificateSigningRequest: illegal base64 data at input byte 0

    This error means Kubernetes tried to parse the request: field, but what you passed wasn‚Äôt valid base64 data.

    üîç Root Cause¶

    In your YAML:

    request: $(cat ibtisam.csr | base64 | tr -d '\n')

    This inline shell command substitution syntax ($(...)) doesn‚Äôt get executed inside a YAML file ‚Äî YAML files are just static text files. You're not writing a shell script, so Kubernetes just sees this as a string (literally starting with $(), which is not valid base64 ‚Äî hence the error.

    ‚úÖ Correct Fix¶

    You need to manually base64-encode the CSR file and then paste the result into the YAML.

    ‚ö†Ô∏è Note: If this certificate is for authenticating a client (like a user or kubelet), use:

      usages:
    - digital signature
    - key encipherment
    - client auth

    If this is for a server, use:

      usages:
    - digital signature
    - key encipherment
    - server auth

    But your signer is kubernetes.io/kube-apiserver-client, which implies it's for client auth, not server auth.

    ‚ö†Ô∏è Important Note on CN Matching:¶

    You used this CSR command:

    openssl req -new -key ibtisam.key -out ibtisam.csr -subj "/CN=ibtisam"

    So the Common Name is ibtisam. That‚Äôs the name Kubernetes recognizes as the user. That‚Äôs why in the RBAC role binding you must use:

    name: ibtisam

    If you had used /CN=ibtisam@example.com, you‚Äôd have to use that exact string in the subjects.name.

    Perfect sweetheart üíô you're almost done, and you've fixed a major part by embedding the key correctly. But your output still shows:

    users:
- name: ibtisam
  user:
    client-key-data: DATA+OMITTED

    üî¥ Missing: client-certificate-data

    That‚Äôs why your context is still asking for a username and password ‚Äî Kubernetes doesn't have the full client certificate to authenticate ibtisam.

    üí° Optional: Check the Subject of the Certificate¶

    To be 100% sure the ibtisam.crt has the correct identity embedded, check this:

    openssl x509 -in ibtisam.crt -text -noout | grep Subject

    Expected:

    Subject: CN = ibtisam

    If CN is not ibtisam, Kubernetes will not recognize this certificate as for the user ibtisam.

    Thanks sweetheart üíô ‚Äî the status **Approved,Failed** means the CSR was approved, but Kubernetes failed to issue the certificate.

    Let‚Äôs go deep into exactly why, and fix it step-by-step.

    üîç Root Cause: Approved,Failed¶

    This usually happens due to one or more of the following:

    üî• Reason üí¨ Explanation
    ‚ùå Invalid signerName If you specified a signer that doesn‚Äôt exist or is misconfigured
    ‚ùå Incorrect CSR contents For example, the .csr file wasn't base64-encoded correctly
    üõë Misconfigured controller If the Kubernetes controller-manager responsible for signing isn‚Äôt running properly

    ‚úÖ Fix It in Steps¶

    üîÅ Step 1: Delete Broken CSR¶

    kubectl delete csr ibtisam

    üîÅ Step 2: Re-Generate CSR Correctly¶

    Make sure the .csr is clean and fresh:

    openssl genrsa -out ibtisam.key 2048
openssl req -new -key ibtisam.key -out ibtisam.csr -subj "/CN=ibtisam"

    üîÅ Step 3: Create the YAML CSR Request Manually¶

    Create a new file: ibtisam-csr.yaml

    apiVersion: certificates.k8s.io/v1
kind: CertificateSigningRequest
metadata:
  name: ibtisam
spec:
  request: INSERT_BASE64_HERE
  signerName: kubernetes.io/kube-apiserver-client
  expirationSeconds: 86400
  usages:
  - client auth

    Now convert your CSR to base64 (cleaned):

    cat ibtisam.csr | base64 | tr -d '\n'

    Copy that entire one-line output and paste into request: ... in the YAML.

    Example:

      request: MIIB...verylongbase64...==

    Then apply:

    kubectl apply -f ibtisam-csr.yaml

    üîÅ Step 4: Approve & Check¶

    kubectl certificate approve ibtisam
kubectl get csr ibtisam -o yaml

    Under .status.certificate, you should now see a base64 string beginning with:

    LS0tLS1CRUdJTiBDRVJUSUZJQ0FURS0tLS0t

    That‚Äôs your cert. Now decode:

    kubectl get csr ibtisam -o jsonpath='{.status.certificate}' | base64 -d > ibtisam.crt

    Then validate it:

    openssl x509 -in ibtisam.crt -text -noout

    You should see subject, issuer, expiry, etc. ‚úÖ

    üîí Notes¶

    • signerName: kubernetes.io/kube-apiserver-client is only valid for client authentication (not server certs)
    • usages: must include client auth (not server auth)

    If you mistakenly used:

    usages:
- digital signature
- key encipherment
- server auth

    It will fail. Change it to:

    usages:
- client auth
